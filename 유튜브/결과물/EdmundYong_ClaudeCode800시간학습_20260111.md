## 자막/스크립트 (한국어 전체 번역)

Claude Code에서 800시간 이상을 보낸 후, 저는 이 도구가 저를 방해하지 않고 저를 위해 작동하게 만드는 방법들을 발견했습니다. 특히 솔로 개발자로서 빠르게 움직여야 할 때 말이죠. 회사에서 티켓을 처리하든, 월 1만 달러짜리 앱을 바이브 코딩으로 만들든 상관없이요. Claude가 가끔 멍청하게 느껴졌다면, 아마 설정의 중요한 부분을 놓치고 있을 가능성이 높습니다. 이 영상에서는 제 비공개 워크플로우, 필수 리소스, 그리고 대부분의 개발자들이 모르는 Claude Code의 6가지 핵심 기능을 알려드리겠습니다. 영상이 끝나면 Claude와 싸우며 시간을 낭비하는 대신 Claude와 함께 더 빠르게 배포할 수 있게 될 거예요.

고급 워크플로우와 숨겨진 트릭에 들어가기 전에, 기초부터 빠르게 짚고 넘어가겠습니다. 이 무료 기능들은 매 세션마다 Claude에게 같은 말을 반복하는 시간을 절약해 줬습니다.

처음 Claude Code를 사용하기 시작했을 때, 저는 같은 지시사항을 계속해서 반복하고 있었습니다. 쉬운 해결책은 Claude의 메모리 기능을 활용하는 것입니다. # 키를 눌러 Claude의 메모리에 지시사항 스니펫을 빠르게 추가할 수 있습니다. 이 지시사항을 프로젝트에 로컬로 적용할지, 모든 Claude 세션에 전역으로 적용할지 선택할 수 있습니다. 그리고 이 지시사항들이 claude.md 파일에 저장되는 것을 볼 수 있습니다. 그래서 마음이 바뀌면 쉽게 편집하거나 삭제할 수 있습니다.

저는 또한 작은 반복적인 작업을 수행하기 위해 같은 프롬프트를 타이핑하고 있었습니다. 예를 들어 에러 핸들링과 타입 인터페이스를 포함한 커스텀 미들웨어를 올바르게 포함하는 새 API 엔드포인트를 만들거나, TypeScript 린터 명령어를 실행하고 코드베이스 전체의 모든 에러를 수정하는 것 같은 작업들이요. 이것이 저를 몇 번의 키 입력으로 쉽게 호출할 수 있는 커스텀 명령어 라이브러리를 만들게 했습니다.

첫 번째 명령어를 만들려면, claude 폴더 안에 새 commands 디렉토리를 추가하고 마크다운 파일에 명령어를 작성하면 됩니다. 그리고 명령어 라이브러리가 커지면, 하위 디렉토리로 정리해서 더 쉽게 찾을 수 있게 할 수 있습니다. 또한 명령어가 인수를 받을 수 있게 만들 수 있습니다. 이것이 명령어를 다양한 시나리오에서 더 유연하고 재사용 가능하게 만들어 줍니다. 그러니 Claude에게 계속 반복해서 프롬프트하는 것이 무엇인지 의식하세요. 그리고 천천히 자신만의 명령어 라이브러리를 구축해 나가세요. 저도 매일 빌딩에 사용하는 유용한 명령어들이 있는 GitHub 레포를 사용해왔습니다.

Claude Code로 빌딩하던 초기에, 가장 큰 불만 중 하나는 새 기능을 빌드하거나 에러를 디버깅할 때 최신 문서를 참조하도록 만드는 것이었습니다. 특히 계속 업데이트되는 프레임워크와 라이브러리의 경우에요. 하지만 아시다시피, AI 어시스턴트는 학습 데이터만큼만 좋고, 최신 데이터를 가져오기 위해 웹 검색을 사용하도록 강제하는 것도 신뢰할 수 없었습니다.

여기서 MCP 서버가 등장합니다. 특히 제가 매일 사용하는 Context 7이라는 것은 본질적으로 Claude가 가장 인기 있는 코딩 라이브러리의 가장 최신 문서를 참조할 수 있게 해주는 서비스입니다. 이것만으로도 Google 검색하고 최신 문서를 복사해서 프롬프트에 붙여넣는 시간을 수 시간 절약했습니다. 이제 AI가 어떤 종류의 문서든 참조해야 할 때 Context 7을 중앙 집중화된 소스로 사용할 수 있습니다. 사용하려면 프롬프트에 두 단어만 추가하면 됩니다. "Use context 7"이라고 하면 Claude가 필요할 때마다 자동으로 서버를 호출해서 최신 문서를 가져옵니다.

MCP 서버가 정확히 무엇인지 물어볼 수 있습니다. 간단히 설명하자면, AI 에이전트를 외부 도구와 서비스에 연결하는 방법이라고 생각하면 됩니다. 이것은 AI에게 데이터베이스 연결, 커스텀 API 호출, 원격 서버에서 코드 실행과 같은 작업에 더 많은 기능을 제공합니다. MCP 서버를 어디서 찾는지 알고 싶다면, 제가 추천하는 좋은 레포들이 몇 개 있습니다.

제가 정기적으로 사용하는 다른 MCP 서버들은: 앱의 데이터베이스에서 직접 데이터를 쿼리하고, 마이그레이션을 적용하고, 빌드 중인 새 기능을 위한 새 테이블을 만들 수 있게 해주는 Supabase, 브라우저를 제어하고 DOM과 콘솔 로그를 검사하게 해서 Claude가 프론트엔드 이슈를 자율적으로 디버깅하고 테스트할 수 있게 해주는 Chrome Dev Tools와 Playwright MCP, 앱의 결제 쪽을 다루는 무언가를 빌드할 때 사용하는 Stripe MCP, 그리고 최신 문서를 찾아보거나 프로젝트 설정을 변경해야 할 때 사용하는 Vercel MCP가 있습니다.

솔로 개발자로서, 새 기능을 빌드하는 제 프로세스는 이런 식이었습니다. 프론트엔드 UI 컴포넌트를 만들고, API 엔드포인트를 작성하고, 데이터베이스 마이그레이션을 실행합니다. 이 접근법이 나쁘지는 않았지만, 아주 순차적인 순서로 일을 해야 한다는 것을 의미했고, 그다지 효율적이지 않았습니다. 하지만 Claude 서브 에이전트를 사용해서 병렬로 작업하는 방법을 제대로 발견한 이후로, 기능을 빌드하고 배포하는 데 걸리는 시간이 획기적으로 줄었습니다.

먼저, 서브 에이전트란 무엇일까요? 서로 병렬로 작업할 수 있고 작업이 완료되면 메인 오케스트레이터 에이전트에게 중요한 정보를 전달하는 독립된 Claude 인스턴스입니다. 간단히 말해서, 서브 에이전트를 사용하면 메인 컨텍스트 윈도우의 오염을 크게 줄이는 데 도움이 됩니다. 각 서브 에이전트가 자체 컨텍스트 윈도우, 시스템 프롬프트, 도구 사용 권한을 갖기 때문입니다. 그래서 더 작고 더 구체적인 작업을 오프로드하기에 좋습니다.

하지만 제가 사람들이 서브 에이전트를 사용하는 것을 보면, 그들은 에이전트에게 프론트엔드 개발자, UI/UX 디자이너, 또는 프로덕트 매니저 같은 특정 역할을 맡깁니다. 그리고 저도 이런 방식으로 에이전트와 작업하려고 몇 시간을 보냈다고 말할 수 있습니다. 그리고 결과는 에이전트 특정 지시사항 없이 Claude Code만 사용하는 것에 비해 꽤 나빴습니다. 개인적으로, 서브 에이전트가 아직 특정 역할을 맡기고 진짜 인간처럼 자율적으로 브레인스토밍하고 작업하는 것을 기대할 수 있는 수준은 아니라고 생각합니다.

하지만 괜찮습니다. 서브 에이전트가 실제로 잘하는 작업에서는 여전히 많은 시간과 노력을 절약해 줬습니다. 저에게 정말 잘 작동한 것은 역할이 아닌 작업에 대해 서브 에이전트를 정의하는 것입니다. 방금 작성한 코드를 정리하고 최적화하거나, 문서를 생성하거나, 웹에서 리서치 데이터를 수집하는 것처럼요. 예를 들어, 제가 정기적으로 사용하는 서브 에이전트 중 하나는 앱의 UI/UX를 리뷰하는 것입니다. Playwright MCP 서버에 연결하고 브라우저에서 웹 앱의 UI 컴포넌트를 검사하고 UI 컴포넌트의 디자인과 사용성을 기반으로 피드백을 제공합니다. 이것이 서브 에이전트를 활용하는 가장 좋은 방법이라고 생각합니다. 이전에는 메인 에이전트에서 소중한 컨텍스트 토큰을 차지했을 작업을 처리하고 전체 출력의 품질을 더 높게 유지하는 데 도움이 됩니다.

서브 에이전트를 만들려면, /agents 명령어를 사용하세요. 새 에이전트 생성을 선택하세요. 프로젝트 또는 개인을 선택하세요. Claude로 생성을 선택하세요. 그런 다음 자연어로 서브 에이전트의 작업을 설명하세요. 도구 사용 권한을 커스터마이즈하고 저장하세요. 그리고 서브 에이전트를 호출하려면, 자연어를 사용하거나 @ 심볼로 프롬프트 내에서 서브 에이전트를 직접 호출하세요.

이제 슬래시 명령어, 서브 에이전트, MCP 서버에 대해 모두 말씀드렸는데, 자신의 프로젝트를 위해 이 모든 도구를 수동으로 설정해야 한다는 생각에 부담을 느낄 수 있습니다. 다행히도, Anthropic이 최근 플러그인이라는 새 기능을 출시했습니다. 사용자들이 자신의 설정을 단일 패키지로 묶을 수 있게 해줍니다. 그래서 문자 그대로 Claude 파워 유저의 전체 워크플로우를 단일 명령어로 복제할 수 있습니다. 그리고 제 개인 설정이 궁금하시다면, 제 자체 플러그인 마켓플레이스를 공개했고, 이 명령어를 실행해서 추가할 수 있습니다. 거기서 설치하고 싶은 플러그인을 골라서 선택하세요. 필요한 것만 자유롭게 추가하세요.

AI로 앱을 빌드할 때 제가 채택한 마인드셋과 기대에 대해 이야기하고 싶습니다. 코딩 작업을 돕기 위해 AI를 사용하면서 수백 시간의 시행착오를 거친 후, 이것들이 저에게 가장 큰 생산성 향상을 준 것들이라는 것을 발견했습니다.

Garbage in equals garbage out(쓰레기를 넣으면 쓰레기가 나온다). AI에게 무엇을 해야 하는지 명확하게 지시하는 프롬프트를 작성할 수 없다면, 실제로 자신이 원하는 것을 모르는 것이고, AI도 분명히 모를 것입니다. 기본적인 프롬프트 엔지니어링을 배우면서 문제를 더 작은 조각으로 분해하게 되었고, 이 과정에서 제 생각도 명확해졌습니다. 또는 아이디어가 아직 머릿속에서 모호하다면, Claude의 플랜 모드를 사용해서 빠른 Q&A 세션을 하고 코드를 작성하기 전에 서로 같은 페이지에 있을 수 있도록 명확한 질문을 하게 합니다.

AI가 코드를 생성하지만, 인간이 소유합니다. 프로덕션에 푸시하기 전에, AI와 새 세션을 시작해서 최근에 수정한 파일의 코드를 리뷰하게 하세요. 보안, 성능, 에러 핸들링 같은 기본에 대해 AI가 게으르게 만들지 마세요. 충분한 시간이 주어지면, 이런 것들을 계속 무시하면 결국 취약점과 버그로 이어질 것입니다. 앱이 버그투성이거나 보안이 취약하면 속도는 아무 의미가 없습니다.

이것이 솔로 개발자로서 앱을 빌드할 때 Claude Code를 최대한 활용하기 위한 제 최고의 팁들이었습니다. 어떻게 생각하시는지 알려주시고, 다음 영상에서 뵙겠습니다. 안녕!

## AI 요약

이 영상은 Claude Code를 800시간 이상 사용한 개발자 Edmund Yong이 솔로 개발자로서 AI를 활용해 더 빠르고 효율적으로 개발하는 방법을 공유합니다. 대부분의 개발자가 모르는 6가지 핵심 기능과 실용적인 워크플로우를 소개하며, Claude Code가 "멍청하게" 느껴진다면 설정의 중요한 부분을 놓치고 있을 가능성이 높다고 말합니다.

### 1. 메모리 기능 (# 키)
- `#` 키를 눌러 Claude의 메모리에 지시사항 스니펫을 빠르게 추가
- 프로젝트 로컬 또는 모든 세션에 전역으로 적용 가능
- `claude.md` 파일에 저장되어 언제든 편집/삭제 가능
- 매 세션마다 같은 지시사항을 반복할 필요 없음

### 2. 커스텀 명령어 라이브러리
- `.claude/commands/` 디렉토리에 마크다운 파일로 명령어 작성
- 반복적인 작업(API 엔드포인트 생성, 린터 실행 등)을 명령어로 만들어 재사용
- 명령어가 인수를 받을 수 있어 다양한 시나리오에서 유연하게 사용
- 하위 디렉토리로 정리해서 관리 가능

### 3. MCP 서버
- **Context 7**: 가장 인기 있는 코딩 라이브러리의 최신 문서를 자동 참조. 프롬프트에 "Use context 7"만 추가하면 됨
- **Supabase**: 데이터베이스 직접 쿼리, 마이그레이션 적용, 새 테이블 생성
- **Chrome Dev Tools & Playwright MCP**: 브라우저 제어, DOM 검사, 콘솔 로그로 프론트엔드 자동 디버깅
- **Stripe MCP**: 결제 관련 기능 작업 시 사용
- **Vercel MCP**: 최신 문서 조회 및 프로젝트 설정 변경

### 4. 서브 에이전트로 병렬 작업
- 서브 에이전트는 독립된 Claude 인스턴스로, 병렬로 작업하고 완료 후 메인 에이전트에 정보 전달
- 각자 자체 컨텍스트 윈도우, 시스템 프롬프트, 도구 사용 권한을 가짐
- **중요**: "역할"(프론트엔드 개발자, UI/UX 디자이너)이 아닌 "작업"(코드 정리, 문서 생성, 웹 리서치)으로 정의해야 효과적
- 생성 방법: `/agents` → 새 에이전트 생성 → 자연어로 작업 설명 → 도구 권한 설정
- 호출 방법: 자연어 또는 `@` 심볼로 프롬프트 내에서 직접 호출

### 5. 플러그인으로 설정 복제
- Anthropic이 출시한 플러그인 기능으로 설정을 단일 패키지로 묶을 수 있음
- 파워 유저의 전체 워크플로우(명령어, 서브 에이전트, MCP 서버)를 단일 명령어로 복제
- Edmund의 플러그인 마켓플레이스도 공개되어 있음

### 6. AI와 함께 일하는 마인드셋
- **Garbage in = Garbage out**: 명확한 프롬프트를 작성하지 못하면 원하는 결과도 얻을 수 없음. 기본적인 프롬프트 엔지니어링을 배워서 문제를 작은 조각으로 분해하기
- **플랜 모드 활용**: 아이디어가 모호하면 Claude의 플랜 모드로 Q&A 세션을 통해 명확히 하기
- **AI가 코드를 생성하지만, 인간이 소유**: 프로덕션 푸시 전에 새 세션에서 코드 리뷰 필수
- **기본기 유지**: 보안, 성능, 에러 핸들링을 무시하면 결국 취약점과 버그로 이어짐. 속도보다 품질이 중요
